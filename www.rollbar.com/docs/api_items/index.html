Note: This documentation is provided for authors of custom integration plugins. If your application is built on one of our supported platforms, you can use a pre-built plugin instead of rolling your own.

Overview

Rollbar accepts Items (errors, warnings, etc.) via a RESTful JSON API. Clients send JSON data via a POST to https://api.rollbar.com/api/1/item/

We strongly recommend using HTTPS, but HTTP is also supported. For HTTP, use http://api.rollbar.com/api/1/item/

POSTed JSON data is accepted either as the body of the request, or form-encoded as the value of the payload parameter (which should be the only parameter).

All responses (including error responses) are returned as JSON.

Example

Here is an example bare-bones request, using Python. Note that the access_token is part of the JSON payload, and that the encoded json payload is sent directly as the request body.

payload = {
  "access_token": "YOUR_PROJECT_ACCESS_TOKEN",
  "data": {
    "environment": "production",
    "body": {
      "trace": {
        "frames": [
          {"filename": "/Users/brian/www/mox/mox/views/project.py",
           "lineno": 26,
           "method": "index",
           "code": "_save_last_project(request, project_id)"
          },
          {"filename": "/Users/brian/www/mox/mox/views/project.py",
           "lineno": 497,
           "method": "_save_last_project",
           "code": "user = foo"
          },
        ],
        "exception": {
          "class": "NameError",
          "message": "global name 'foo' is not defined"
        }
      }
    }
  }
}

json_encoded_payload = json.dumps(payload)
requests.post('https://api.rollbar.com/api/1/item/', data=json_encoded_payload)

If your library does not allow you to specify the raw POST body, you can send it form-encoded as the value of the payload parameter. For example, using Python:

requests.post('https://api.rollbar.com/api/1/item/', data={'payload': json_encoded_payload})

Error Responses

The Items API can return the following status codes:

200
Success. The item was accepted for processing.
400
Bad request. No JSON payload was found, or it could not be decoded.
403
Access denied. Check your access_token.
422
Unprocessable payload. A syntactically valid JSON payload was found, but it had one or more semantic errors. The response will contain a "message" key describing the errors.
429
Too Many Requests - If rate limiting is enabled for your access token, this return code signifies that the rate limit has been reached and the item was not processed.
500
Internal server error. There was an error on Rollbar's end.

If you encounter any other errors, or don't understand an error you see, please contact support.

Data Format

The top-level payload contains two keys, access_token and data, both of which are required: 

{
  "access_token": "YOUR_PROJECT_ACCESS_TOKEN",
  "data": {
    ...
  }
}

access_token is the access token for the project the item will be submitted to.

data is a JSON object with two required keys, environment and body, and many optional keys. The optional keys listed below have special meaning within Rollbar and are specifically supported. Other data may be passed as well and will be stored.

environment
Required. The environment name, e.g. "production", "staging", "development". Environment names are free-form strings and can be up to 255 characters long. For best results, use short names.
body
Required. The body of the item. Described in detail in the next section.
timestamp
A UNIX timestamp (in seconds since the unix epoch) indicating when the item occurred. If omitted, the time when the item is received will be used.
level
The severity level of the item. One of "critical", "error", "warning", "info", or "debug".
platform
A string describing the platform on which the code was run. Can be used to filter items in the Rollbar UI. e.g. "browser" or "flash" or "heroku" or "google-app-engine"
language
The name of the code's language (e.g. "ruby" or "actionscript3" or "python")
framework
The name of the code's framework (e.g. "rails")
request
An object describing the request associated with the item. Described in detail below
person
An object describing the user affected by the item. Described in detail below
server
An object describing the server environment. Described in detail below
client
An object describing the client environment. Described in detail below
custom
An object containing arbitrary custom data.
notifier
An object describing the notifier plugin. Described in detail below
fingerprint
Advanced: a string up to 40 characters long that identifies the "fingerprint" of this item. Items with the same fingerprint are grouped together in Rollbar. Rollbar will automatically compute a fingerprint, but you can pass one explicitly if you want to override it.
title
Advanced: a string up to 255 characters that will be used as the item title in the Rollbar UI. Rollbar will automatically compute a title, but you can pass one explicitly if you want to override it.

body is a JSON object describing the item itself. It should contain either a trace key or a message key (but not both).

If body contains a message key, it must be an object itself containing a body key. It can also contain additional keys containing arbitrary data. The value of the body key will be used to determine the title and will be used as the main message body. Additional keys will appear in the Occurrences table.

If body contains a trace key, it must be an object describing a traceback. The trace must contain a frames list and exception object.

The frames list should be ordered such that the most recent call is at the end of the list. Each frame must contain a filename key and may contain several other keys:

filename
Required. The filename including the path.
lineno
The line number
method
The method or function name
code
The line of code
context
An object containing pre and post code context, as lists of lines of code.

The exception object must contain a class key and may contain a message key:

class
The class name of the exception (or some other string describing the error class)
message
The exception message (should not be prefixed with the class name)

request is a JSON object describing the request that triggered the item being reported. Rollbar understands the following keys:

url
The full request URL
method
The request method (e.g. "GET")
headers
A JSON object containing the request headers (cookies, user-agent, etc.)
params
A JSON object containing any routing paramters (e.g. for use with Rails Routes)
GET
A JSON object containing any GET params
query_string
A string containing the raw query string
POST
A JSON object containing any POST params
body
A string containing the raw POST body
user_ip
The user's IP address as a string. Can also be the special value "$remote_ip", which will be replaced by the source IP of the API request (use this for client-side errors).

person is a JSON object describing the user affected by the item. 'id' is required.

id
(required) User ID of the affected user. A string up to 40 characters.
username
Affected user's username. A string up to 255 characters. Optional.
email
Affected user's email address. A string up to 255 characters. Optional.

server is a JSON object describing the server environment for the server associated with the item being reported. Rollbar understands the following keys:

host
The server hostname, e.g. "web1" or "web1.mysite.com"
root
The path to the application code root, not including the final slash
branch
The name of the checked out source control branch. Defaults to "master"
log_file
The name of the log file the item was found in

client is an arbitrary JSON object describing the client environment. As there can be multiple client environments embedded within each other (i.e. Flash running inside an HTML page that also has javascript), data should be namespaced by platform. For example:

{
  ...
  "data": {
    ...
    "client": {
      "javascript": {
        "browser": "user agent string here",
        ...
      },
      "flash": {
        "version": "11.2",
        "capabilities": {
          ...
        }
      }
    }
  }

}

notifier is a JSON object describing the notifier that reported this item (i.e. the plugin you are writing). It should contain the following keys:

name
A string identifying the notifier
version
The notifier version, as a number or string
